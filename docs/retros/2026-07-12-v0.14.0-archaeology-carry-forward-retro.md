## Key Findings

### 1. State tables made a complex retry contract implementable

The approved spec enumerated eight combinations of `--pr-bodies`, token availability, and prior processing state. Task implementation and regression tests could therefore cover tokenless patch extraction, credentialed enrichment, empty PR bodies, and hard API failures without inventing semantics during coding.

**Why:** Patch completion and PR-body completion were modeled as separate monotonic facts instead of one overloaded processed bit. The explicit FOUND/EMPTY/FAILURE fetch contract also prevented empty bodies from becoming permanent retries.

**How to apply:** Keep state-transition tables in specs whenever persistence and retry behavior interact. Require every row to map to an automated assertion before implementation ends.

### 2. Two-level review caught integration invariants that task reviews missed

Task reviews each converged after one fix round, but final branch review still found three Important integration defects: tokenless PR-only counter handling, EMPTY-result completion gating, and a negative live denominator. All were fixed before PR creation.

**Why:** Each task test proved its local contract, while counters combined processing state, fetch outcomes, and progress reporting across task boundaries. No single task initially owned the conservation invariant.

**How to apply:** Retain both task and final branch review. For progress counters, assert conservation directly: every selected commit must land in exactly one terminal or retryable bucket, and no displayed count may become negative.

### 3. Faithful process doubles matter more than broad mock coverage

The Task 4 reviewer rejected a fake `Popen` stdout that returned arbitrary chunks unlike `TextIOWrapper.readline()`. Replacing it with a faithful line-oriented double made the record-boundary and early-close tests prove the actual subprocess contract.

**Why:** The first double exercised a more permissive interface than production, so passing tests could not establish behavior at real line boundaries.

**How to apply:** Model the narrowest real protocol in test doubles, especially for streaming I/O. A double should preserve framing, EOF, cleanup, and lazy-consumption semantics rather than merely deliver representative bytes.

### 4. Measure-first coverage exposed a release-contract omission locally

All nine design metrics mapped to automated tests. The full suite then caught that the CHANGELOG still described schema v16 after the migration moved to v17; the omission was fixed before GitHub CI, which completed with zero failures.

**Why:** Targeted archaeology tests verified runtime behavior, but repository contract synchronization was owned by a separate existing test module.

**How to apply:** Continue running changed-module tests first, then the full contract suite before shipping. Treat documentation/schema synchronization as part of the measurable feature contract, not release polish.

### 5. Resume state worked, but release lineage needed isolation

The loop resumed after interruption using `.release-loop/progress.md` and git evidence. During Ship, local `main` also contained earlier release-loop skill commits, so pushing it would have published unrelated history. A clean `release/v0.14.0` branch from `origin/main` carried only the release commit and this retro.

**Why:** The feature was squash-merged while local `main` had independent commits not present in the PR, making branch ancestry unsuitable for a direct release push.

**How to apply:** Trust tracked progress when resuming, but verify remote ancestry before release commits. If local base history diverges, release from a clean `origin/main` lineage and cherry-pick only intended commits.

## Lessons

**"Counter conservation is a cross-task contract"** — three Important final-review fixes showed that locally correct state transitions can still produce globally invalid progress counts; assert terminal-bucket conservation explicitly.

**"Streaming tests are only as strong as their framing model"** — a fake stdout that emitted arbitrary chunks was replaced because production `TextIOWrapper` is line-oriented; protocol-faithful doubles prevent false confidence.

**"A clean release lineage is safer than repairing unrelated local history"** — releasing from `origin/main` plus the intended release commit avoided pushing three local-only skill commits.
