## Key Findings

### 1. Release-loop skill completed its first full cycle successfully

All 6 phases (Design → Plan → Implement → Review → Ship → Retro) ran to completion in a single session. The skill orchestrated 4 subagents in parallel, managed review feedback, and produced a merged PR with zero CI failures.

**Why:** The skill's state tracking (`progress.md`) and phase transition protocol worked as designed. Each phase reference doc provided enough structure to avoid ad-hoc decisions.

**How to apply:** Keep the release-loop skill as-is for patch releases. For larger features, consider whether the single-session constraint is realistic or if multi-session state persistence needs hardening.

### 2. Independent review caught a real runtime bug that implementation + tests missed

Branch review found that `proc.terminate()` was unconditional in the `finally` block, causing `returncode=-15` (SIGTERM) on successful runs — producing spurious "git log failed" warnings. The implementation passed all tests because the warning path doesn't affect yielded data.

**Why:** Tests verified output correctness but not side-effect cleanliness (warning absence on success). The bug was invisible at the functional level.

**How to apply:** For process-management code, add a negative assertion: "no warnings on clean exit". The branch review's value proposition (catching bugs that pass tests) is confirmed — don't skip it for "simple" features.

### 3. Plan review prevented implementation rework (again)

The Fable plan review found 1 CRITICAL API mismatch (test code used wrong `persist_candidate`/`score_confidence` signatures) + 4 IMPORTANT issues (branch name assumption, missing callers, missing imports, placeholders). Without this review, Task 1 tests would have failed for the wrong reasons, wasting debugging time.

**Why:** Plan code examples were written from memory rather than verified against actual source. The plan reviewer reads source files and catches drift.

**How to apply:** Plan review before implementation is mandatory — third consecutive release (v0.13.0, v0.13.1) confirming this. The cost is ~3 min of review time; the saving is 10-30 min of debugging wrong test scaffolding.

### 4. Dual release (v0.13.0 tag + v0.13.1 feature) in one session is viable but creates rebase friction

v0.13.0 had been merged but never tagged. Tagging it first, then developing v0.13.1 on a feature branch, then squash-merging back created a rebase divergence on main (local had spec/plan commits that were inside the squash). Required `git rebase --skip` to resolve.

**Why:** The squash merge absorbs the spec/plan commits into one PR commit, but they still exist locally as separate commits on main (committed before branching). The divergence is mechanical, not semantic.

**How to apply:** When doing a tag-then-feature in one session, branch off *before* committing spec/plan to main, or commit spec/plan directly to the feature branch.

## Lessons

**"Branch review catches runtime bugs that pass functional tests"** — `proc.terminate()` producing `returncode=-15` warnings was invisible to output-correctness tests. Side-effect assertions (no warnings on success path) are needed for process management code.

**"Plan review ROI is proven across 3 consecutive releases"** — v0.13.0 caught pipeline fork risk + FTS trigger; v0.13.1 caught API signature mismatch + missing callers. 3 min review → 10-30 min saved debugging. Mandatory.

**"Same-session dual release works but creates rebase noise"** — commit spec/plan to the feature branch (not main) when releasing and developing in one session to avoid post-squash divergence.
