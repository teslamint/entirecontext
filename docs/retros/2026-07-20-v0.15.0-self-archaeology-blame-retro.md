## Findings

### What worked well

- **What happened**: U5 processed 334/334 eligible feature-branch commits, promoted 138 archaeology candidates, and moved file-link coverage from 8.6% to 56.0% (`.release-loop/evidence/U5/step5-post-confirm-metrics.txt`).
  **Why**: resumable archaeology plus a dry-run-first 0.35 threshold made the large bootstrap observable before mutation.
  **How to apply**: retain the dry-run distribution and before/after coverage queries for future corpus migrations.

- **What happened**: four ready-for-review rounds produced nine fixes covering SHA-256 porcelain, binary text, mixed ranges, schema migration, abbreviated and uppercase SHAs, deduplication, non-finite thresholds, and migration documentation; final CI and the merged full suite were green.
  **Why**: the branch was moved from draft to Ready and review continued until the configured review cap, rather than treating the first green build as sufficient.
  **How to apply**: keep review readiness as an explicit gate and preserve each finding as a regression test when it exercises a concrete code path.

- **What happened**: the repository-local `release-loop` implementation was removed without deleting historical `.release-loop` evidence or the user's unrelated `LESSONS.md` and experiment-log changes.
  **Why**: cleanup validation and git operations were scoped to the migration paths and worktrees.
  **How to apply**: separate tracked migration deletions from runtime history and unrelated dirty-tree state in future repository cleanup.

### What to improve

- **What happened**: the U3/U4 plan missed enough real porcelain and commit-link edge cases to require repeated review fixes, and one SQLite expression-depth P2 remained after the review cap.
  **Why**: the initial matrix covered standard SHA-1 porcelain and small commit sets, but not alternate object formats, non-UTF-8 blobs, equivalent abbreviated links, non-finite CLI floats, or very high distinct-SHA counts.
  **How to apply**: add an adversarial boundary matrix before implementation review, including object format, encoding, canonicalization, numeric finiteness, and storage-engine limit cases.

- **What happened**: the declared full-history criterion compared the raw `archaeology_processed` count with eligible commits. After squash merge, the table had 334 rows while reachable history had 326 commits and one new pending squash SHA.
  **Why**: processed branch SHAs persist even when the branch becomes unreachable, while the squash merge creates a new reachable commit.
  **How to apply**: compare the intersection of reachable eligible SHAs and processed SHAs, and re-run convergence after squash merge. The reusable guidance is documented at `docs/solutions/workflow-issues/measure-archaeology-against-reachable-history.md`.

- **What happened**: the prior TQL retro named two P3 gaps but `ROADMAP.md` did not contain them.
  **Why**: the previous retrospective predated the current carry-forward reconciliation contract and its final table was not checked against the roadmap.
  **How to apply**: make retro closure fail when the previous carry-forward set and current roadmap set do not reconcile; both gaps are now registered under v0.15.0 carry-forward.

### Process observations

- **What happened**: the longest unplanned segment was Ready-state review, not implementation: four rounds expanded nine concrete findings across the same small code surface.
  **Why**: the plan optimized for functional scenarios, while reviewers exercised representation boundaries and database limits.
  **How to apply**: budget a boundary-focused review round separately from functional unit review for git/SQLite integration work.

- **What happened**: a fresh retrospective measurement contradicted the expected 0-pending result even though U5 evidence was valid at merge readiness.
  **Why**: measurement time and git topology were implicit in the success criterion.
  **How to apply**: every history-sensitive success criterion should state both the ref and lifecycle point being measured.

- **What happened**: the accepted P2 was not left only in a PR thread.
  **Why**: the user explicitly accepted the residual for merge, and the retrospective carry-forward rule required durable tracking.
  **How to apply**: convert every accepted review residual into a roadmap item with type, priority, and a regression-shaped completion condition.

## Lessons

- **"Full-history completion is a reachable-set property, not a row-count equality."** A squash merge can add one pending reachable commit while leaving many processed branch SHAs unreachable.
- **"Review residuals become real debt only when their completion condition survives the PR."** The accepted SQLite-depth P2 now names both the failure boundary and required regression proof in `ROADMAP.md`.
