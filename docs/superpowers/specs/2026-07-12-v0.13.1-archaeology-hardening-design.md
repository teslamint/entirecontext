# v0.13.1 Archaeology Hardening Design

_Created 2026-07-12. Revised after independent review._

## Overview

Patch release hardening the Git Archaeology feature shipped in v0.13.0. Fixes the `decision_commits` linkage gap for archaeology candidates, adds streaming git-log processing for large repos, skips merge commit messages from extraction, and updates CLI help text for source-type filtering.

## Scope

### In

1. **`decision_commits` linkage on archaeology candidate promotion** — when `confirm_candidate()` promotes an archaeology candidate (`source_type='archaeology'`, `checkpoint_id=None`), use `source_id` (commit SHA) to create a `decision_commits` row. Currently skipped because the linkage path requires `checkpoint_id`.
2. **Streaming Popen for archaeology** — replace `subprocess.run()` + full stdout capture in `_stream_commits()` with `Popen` streaming, AND restructure `archaeologize()` to consume the generator lazily instead of `list()`. Memory usage stays constant regardless of repo size.
3. **Merge commit exclusion** — add `--no-merges` to the git log command. Note: `git log --patch` with default `--diff-merges=off` already suppresses merge diffs, so this only skips merge commit *messages* (e.g. "Merge pull request #NNN") from reaching the extraction pipeline — preventing low-value candidates from merge boilerplate.
4. **`--source` CLI help update** — add `archaeology` to the `--source` option help text in `ec decision candidates list` (`decisions_cmds.py:705`) and MCP tool parameter description.
5. **ROADMAP creation** — create `docs/ROADMAP.md` with v0.13.0 carry-forward items and v0.14.0 candidates.

### Out

- Schema changes (none needed — still v16)
- Tokenless `--pr-bodies` re-run (deferred)
- `dir b/name` path parsing (P4, deferred)
- Source column in candidate listing (already implemented: `decisions_cmds.py:730,737`)

## Architecture

### 1. `decision_commits` linkage fix

File: `src/entirecontext/core/decision_candidates.py`, `confirm_candidate()` (~line 187-201)

After the existing `checkpoint_id` block, add an `elif` for archaeology:

```python
elif candidate.get("source_type") == "archaeology" and candidate.get("source_id"):
    source_id = candidate["source_id"]
    if len(source_id) in (40, 64) and all(c in "0123456789abcdef" for c in source_id.lower()):
        try:
            link_decision_to_commit(conn, decision_id, source_id)
        except Exception:
            pass
```

The hex-length check (40 for SHA-1, 64 for SHA-256) guards against non-SHA `source_id` values from future source types. Runs inside the existing `transaction()` block, so rollback on Step 2 failure still applies.

### 2. Streaming Popen

Two changes required:

**a) `_stream_commits()` — `archaeology.py`**

Replace `subprocess.run()` with `subprocess.Popen()`:

```python
proc = subprocess.Popen(
    cmd, cwd=repo_path,
    stdout=subprocess.PIPE, stderr=subprocess.PIPE,
    text=True, errors="surrogateescape",
)
```

- Read stdout line-by-line (or in chunks), accumulate into record buffer, split on `\x1e` record separator. Yield complete records; buffer incomplete trailing record until next `\x1e` or EOF.
- **Deadlock prevention**: stderr pipe can fill (64KB buffer) and block git. Drain stderr to a temporary `io.StringIO` via a daemon thread, or use `stderr=subprocess.DEVNULL` since git-log stderr is only useful for the initial error check. Prefer the thread approach to preserve warning messages.
- **Generator cleanup**: wrap in `try/finally` to call `proc.terminate(); proc.wait()` if the caller abandons iteration early (exception, KeyboardInterrupt). Without this, orphaned git processes leak.
- **Behavioral change**: current `subprocess.run()` is all-or-nothing (zero commits on git failure). Streaming yields partial results before discovering failure. This is acceptable — each yielded commit is independently valid, and `_is_processed()` dedup prevents double-processing on retry.

**b) `archaeologize()` — caller restructure (`archaeology.py:252`)**

Current code: `commits = list(_stream_commits(...))` materializes everything.

Change to single-pass lazy consumption:
- `commits_scanned` counter: increment per yield
- `already_processed` counter: check `_is_processed()` per commit as it arrives
- `dry_run` path: iterate the generator to count, don't store commits
- `_process_batch()` path: pass commits directly without pre-collecting

This eliminates the O(repo) memory allocation.

### 3. Merge commit exclusion

File: `src/entirecontext/core/archaeology.py`, `_stream_commits()`

Add `"--no-merges"` to the `cmd` list. This skips merge commits entirely (no message, no diff output for merges). The rationale is not "duplicate diffs" (diffs are already suppressed by default), but avoiding merge boilerplate messages like "Merge pull request #123 from user/branch" from generating low-value extraction candidates.

## Risks

| Risk | Mitigation |
|------|-----------|
| Streaming parser mishandles records split across chunk boundaries | Buffer incomplete records; add boundary test with synthetic multi-chunk data |
| Popen deadlock from stderr pipe filling | Drain stderr via daemon thread or StringIO |
| Generator cleanup — orphaned git processes | `try/finally` with `proc.terminate(); proc.wait()` |
| Partial results on git failure (behavioral change from all-or-nothing) | Acceptable — each commit is independently valid; `_is_processed()` dedup prevents double-processing |
| `--no-merges` skips merge commits with unique content | Unlikely in practice; merge messages are boilerplate. Reversible if needed. |
| Archaeology `source_id` might not be a valid SHA in future | Hex-length validation guard (40/64 chars) prevents false linkage |
